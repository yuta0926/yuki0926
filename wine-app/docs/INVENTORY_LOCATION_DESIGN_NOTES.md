# 在庫ロケーション管理の再設計 検討メモ

最終更新: 2026-07-18
ステータス: **未着手 / クライアント確認待ち**(実装前に方針確定が必要)

このドキュメントは、入出庫・移動ロジックに関する設計課題の検討メモです。実装方針が固まるまでのたたき台として残します。関連: `ROADMAP.md`, `docs/SPEC.md` 3.3節。

## 1. 課題

現状、ワイン1レコードは「保管場所(`location`)1つ + 合計在庫数(`quantity`)1つ」しか持たない。そのため以下のような不整合が起きる。

- 横浜で4本入庫 → 追加で東京へ3本入庫すると、実在庫は横浜4本・東京3本のはずが、画面上は「7本 横浜」と表示される。
- 入庫時に保管場所が未入力でも登録できてしまう。
- `move`(移動)は「現在庫を丸ごと別の場所へ付け替え」という設計(`routers/wines.py` の `create_wine_transaction`、コメントに「保管場所は1本ごとではなくワイン単位で管理」と明記)であり、部分移動や複数拠点の同時保有を想定していない。

根本原因: `wines.location` が単一文字列カラムであり、`in`/`out` トランザクションが場所を無視して `wines.quantity` を合算するだけになっている(`backend/app/routers/wines.py:701-729`、`docs/SPEC.md` 3.3節)。

## 2. 検討パターン

### パターンA: ロケーション別在庫の内訳テーブルを追加(1ワインレコードは維持)

`wine_location_stocks`(`wine_id` + `location` + `quantity`)を新設し、1つのワインレコードが複数ロケーションに分散した在庫を持てるようにする。`wines.quantity` は内訳の合計値として維持。

**懸念事項:**
- 拠点ごとに売価・仕入値が異なる場合、価格情報をどこに持たせるか別途設計が必要(ワイン本体は1レコードのままなので、価格はロケーション横断で共通という前提になってしまう)。
- 一覧・詳細・検索・Excelインポートなど、`location` を単一値として扱っている既存箇所すべてに影響(表示は「代表ロケーション+件数」か「複数行展開」かの UI 設計が別途必要)。
- 出庫・移動時に「どのロケーションから」を必ず指定させる必要があり、UIの入力ステップが増える。
- 在庫の合計値(`wines.quantity`)と内訳テーブルの合計を常に一致させる整合性担保(トランザクション境界)が必要。

### パターンB: 1ワインレコード = 1ロケーション(レコード自体を拠点ごとに分割)

同じワイン(銘柄・生産者・Vintage・サイズ等)でも、保管場所が異なれば別レコードとして登録する。拠点ごとに売価・仕入値・数量が独立して持てる。

**懸念事項:**
- 「同じワインが複数拠点に存在する」ことをアプリ上でどう束ねて見せるか(一覧で別々の行になり、検索性・見やすさが変わる)。現状の重複チェック(`name + producer + vintage + size + location`)は既に `location` を含んでいるため、この方針とは元々親和性が高い。
- 移動(`move`)の意味が変わる。現状の「ワインAのロケーションを書き換える」ではなく、「拠点Xのレコードから在庫を減らし、拠点Yのレコードへ加算(存在しなければ新規作成)」という、実質2レコードにまたがる操作になる。
- 拠点をまたいだ在庫合計(このワインは全社で何本あるか)を見たい場合、複数レコードを集計するクエリ・画面が別途必要になる。
- Excel一括登録・重複チェック・編集フォームなど、既存の「ワイン=1エンティティ」という前提で書かれたロジック全般の見直しが必要(影響範囲がパターンAより広い可能性)。

### 比較の軸(クライアント確認時に使う)

| 観点 | パターンA(内訳テーブル) | パターンB(1レコード1拠点) |
|---|---|---|
| 拠点ごとに価格が異なる運用 | 別途価格テーブルが必要 | レコードが分かれるので自然に表現できる |
| 実装影響範囲 | 中(新テーブル+ロジック変更) | 大(重複チェック・一覧・編集・インポート全般に波及) |
| 「このワイン全社で何本?」の見やすさ | 内訳テーブルを合算すればよい | 複数レコードを横断集計する画面が別途必要 |
| 移動操作の実装 | 内訳テーブルの行を減算/加算 | レコードをまたいだ加減算(存在しなければ新規作成) |

## 3. クライアントに確認すべき論点

1. **拠点ごとに売価・仕入値が異なるか。** ここが分岐点。異なるならパターンB寄り、共通ならパターンA寄り。
2. 入庫時に保管場所を必須にしてよいか(現状は未入力可)。
3. 移動を「全量移動」のままでよいか、「部分移動」(在庫の一部だけを別拠点へ)が必要か。
4. 保管場所の表記ゆれ対策(自由入力のままか、既存ロケーションからの選択式にするか)。
5. 1ワインが複数拠点にまたがる場合、一覧・検索・Excelインポートでどう見せたいか。
